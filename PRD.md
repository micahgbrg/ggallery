# Product Requirements Document: Museum Virtual Sculpture Gallery

**Project Codename:** Gallery  
**Version:** 0.1 (MVP)  
**Last Updated:** April 16, 2026  
**Author:** Micah (GBRG Growth Lab)

---

## 1. Problem Statement

Museums worldwide have invested heavily in photogrammetry — producing high-fidelity 3D scans of their sculpture collections. The Smithsonian alone has thousands of open-access 3D models. However, there is no open-source tool that allows a museum to turn a folder of 3D sculpture files into a navigable, spatial virtual gallery experience on the web.

Existing solutions either display single objects in isolation (Smithsonian Voyager, Google `<model-viewer>`, Sketchfab embeds) or require hiring a web developer to build a bespoke Three.js experience from scratch. The gap is the middle layer: an opinionated, configurable, deploy-ready gallery that non-technical museum staff can set up by editing a YAML file and dropping in GLB assets.

## 2. Product Vision

An open-source GitHub template repository that any museum can clone, configure via YAML, populate with GLB/glTF sculpture files, and deploy as a navigable 3D virtual gallery to any static hosting provider (Netlify, Vercel, GitHub Pages, institutional servers).

The gallery is not a model viewer grid. It is a spatial experience — visitors walk through a virtual room and encounter sculptures on pedestals, with museum-quality lighting, informational labels, and accessible navigation.

## 3. Target Users

### Primary: Museum Digital Staff
- Have GLB/glTF files from photogrammetry workflows
- Comfortable editing YAML and running `npm` commands (or can follow a step-by-step README)
- Do not write JavaScript/TypeScript
- Need to deploy to institutional hosting or free-tier platforms

### Secondary: Web Developers Working for Museums
- Contracted to customize the gallery experience
- Comfortable modifying React/Three.js code
- Need a solid, well-documented foundation to build on rather than starting from scratch

### Tertiary: Open-Source Contributors
- Three.js / R3F developers interested in cultural heritage tech
- Want to add room templates, features, or integrations

## 4. Core User Stories (v0.1 MVP)

| ID | As a... | I want to... | So that... |
|----|---------|-------------|-----------|
| U1 | Museum staffer | Clone the repo, add my GLB files and a YAML config, and run `npm run build` | I get a deployable static site with a virtual gallery |
| U2 | Museum visitor | Open the gallery URL on my laptop and navigate between sculptures by clicking viewpoints | I can explore the collection without installing anything |
| U3 | Museum visitor | Click on a sculpture and read its title, artist, date, medium, and description | I learn about the artwork as I would in a physical museum |

## 5. Functional Requirements

### 5.1 Configuration System
- **FR-1:** Gallery is configured entirely via a single `gallery/gallery.yaml` file
- **FR-2:** Config schema includes: gallery metadata (title, institution, description, logo), template selection, theme presets, sculpture list with per-item metadata, navigation mode, and accessibility flags
- **FR-3:** Sculptures are referenced by file path relative to the `gallery/` directory
- **FR-4:** All config fields have sensible defaults; only `sculptures[].file` and `sculptures[].title` are required
- **FR-5:** Config is validated at build time with clear error messages for missing/invalid fields

### 5.2 Gallery Rendering
- **FR-6:** The gallery renders a 3D room using a selected template (v0.1: "long-hall" only)
- **FR-7:** Sculptures are placed on pedestals in positions determined by the template layout algorithm
- **FR-8:** The room includes walls, floor, ceiling, and ambient + accent lighting
- **FR-9:** Each sculpture with `spotlight: true` receives a dedicated directional light
- **FR-10:** The gallery displays the institution name/logo in the UI overlay

### 5.3 Navigation
- **FR-11:** Default navigation is waypoint-based: predefined camera positions with smooth animated transitions
- **FR-12:** Waypoints are auto-generated by the template to provide a good viewing angle for each sculpture
- **FR-13:** Visitors navigate between waypoints via clickable floor markers and keyboard arrows
- **FR-14:** A minimap or navigation menu shows all waypoints and current position
- **FR-15:** Navigation works on both desktop (mouse/keyboard) and mobile (touch)

### 5.4 Sculpture Interaction
- **FR-16:** Clicking/tapping a sculpture opens an info panel overlay
- **FR-17:** The info panel displays: title, artist, date, medium, description, and an optional image
- **FR-18:** The info panel can be dismissed by clicking outside it or pressing Escape
- **FR-19:** While the info panel is open, the sculpture can be orbited (OrbitControls on the selected object)

### 5.5 Asset Pipeline
- **FR-20:** `npm run optimize` processes all GLB files referenced in `gallery.yaml`
- **FR-21:** Optimization includes: mesh simplification (configurable triangle target), texture compression (WebP), and Draco/meshopt compression
- **FR-22:** Original files are preserved; optimized versions are written to `.cache/optimized/`
- **FR-23:** The build reads from `.cache/optimized/` if available, otherwise from source

### 5.6 Build & Deploy
- **FR-24:** `npm run dev` starts a local development server with hot reload
- **FR-25:** `npm run build` produces a self-contained `dist/` folder deployable to any static host
- **FR-26:** The build output works with relative paths (no hardcoded base URL by default)
- **FR-27:** The repo includes a GitHub Actions workflow for deploying to GitHub Pages

## 6. Non-Functional Requirements

| Category | Requirement |
|----------|------------|
| Performance | Gallery loads and becomes interactive within 5 seconds on a 50 Mbps connection with 5 sculptures under 20MB total |
| Performance | Maintains 30+ FPS on a 2020-era mid-range laptop |
| Accessibility | All sculptures reachable via keyboard navigation |
| Accessibility | Screen reader announces sculpture titles at each waypoint |
| Accessibility | `reduced_motion` config flag disables camera animations |
| Compatibility | Works in current versions of Chrome, Firefox, Safari, Edge |
| Compatibility | Responsive layout works on screens from 375px to 2560px wide |
| Bundle Size | Production JS bundle (excluding GLB assets) under 300KB gzipped |
| Hosting | No server-side runtime required; pure static files |

## 7. Scope Boundaries

### In Scope for v0.1
- One room template ("long-hall")
- Waypoint navigation
- Sculpture metadata display
- GLB/glTF loading with Drei's `useGLTF`
- Optimization script using glTF-Transform
- Responsive desktop + mobile
- Sample gallery with CC0 Smithsonian models

### Explicitly Out of Scope for v0.1
- Free-roam WASD navigation (v0.2)
- Additional room templates (v0.2)
- Audio guide support
- Multilingual labels
- VR/WebXR support
- CMS or visual editor
- User analytics
- Server-side rendering
- Gaussian splatting format support

## 8. Tech Stack

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Framework | React 18+ | Component model, ecosystem, Suspense for loading |
| 3D Rendering | Three.js via `@react-three/fiber` | Declarative 3D, large ecosystem |
| 3D Utilities | `@react-three/drei` | Pre-built controls, loaders, HTML overlay |
| Build Tool | Vite | Fast builds, good static output, native TS support |
| Language | TypeScript | Type safety for config schema, contributor DX |
| Config Parsing | `yaml` (npm) | Zero-dependency YAML parser |
| Asset Optimization | `@gltf-transform/cli` + `meshoptimizer` | Industry-standard glTF pipeline |
| Styling | Tailwind CSS | Utility classes for UI overlay, no CSS-in-JS overhead |

## 9. Success Metrics

For the open-source project (not a product with analytics):

- **Adoption:** 50+ GitHub stars within 3 months of launch
- **Usability:** A non-developer can go from clone to deployed gallery in under 30 minutes following only the README
- **Quality:** Zero critical rendering bugs across Chrome, Firefox, Safari on the sample gallery
- **Community:** At least 2 external contributors submit a PR within 6 months

## 10. Open Questions

1. **Navigation UX:** Should waypoint transitions use a smooth camera dolly or a quick fade-to-black teleport? Smooth looks better but can cause motion sickness. Needs user testing.
2. **Mobile performance:** Large photogrammetry meshes may overwhelm mobile GPUs even after optimization. Should the mobile experience fall back to a simpler 2D card layout with `<model-viewer>` per sculpture?
3. **Smithsonian compatibility:** Should the YAML schema align with the Smithsonian 3D Metadata Model for interoperability? This adds complexity but could drive adoption.
4. **License:** MIT is the most permissive and standard for templates. Apache 2.0 provides patent protection. Needs a decision before launch.
